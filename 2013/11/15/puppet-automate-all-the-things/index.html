Puppet: Automate All The Things Everything As Code

November 15, 2013

Updating infrastructure is difficult. Especially when you want to move from an old paradigm of manual everything to automated everything. Configuration Management is something I’ve been hearing about more and more, but I haven’t had a chance to dig into it. With the go ahead to build some new infrastructure, I’m now diving deep into Puppet.

I dislike idealogical fights over tools. The process I try to follow is:

  1. Determine what is needed
  2. Research and decide which tool best meets those needs.
  3. Choose one.
  4. If the one you picked doesn’t work, try another.
  5. Repeat until you’ve either exhausted all the options or found one that works.

I enjoy diving into things and getting stuff done, but there’s a reason why determining needs is the first step. It’s important to know what’s needed, this is the time to design the system that suits the situation best. Things don’t move forward if there is a reliance on the tools that exist and their capabilities determine the needs. After it’s been determined what’s needed, then the available tools are evaluated. If nothing fits the lowest end of the criteria, either build something new or make compromises. Keep those initial needs somewhere important because products grow and suggestions are usually welcome (especially in open source projects).

I did this evaluation and figured out I have resources to learn puppet and it would work with the infrastructure we choose. If puppet doesn’t work, we’ll give Chef a try and then pick between the two.

Learning Resources

One of the factors in the decision to use Puppet was the available resources for to learn it. I have an account on Tutsplus and a few months ago they made the Puppet 3 Beginner’s Guide available as a free download for anyone with an account. If the book had been a Chef book instead, Chef might be in the title of this post instead.

I’ve gotten halfway through the book and I’m excited to get started with a real project. The book focuses on using practical examples and walking through the thought process of a beginner. It gives a fair survey of the landscape as well. It’s a great book for a beginner.

An example of this is a section which discusses distributing manifests. Instead of a Puppet Master, the book suggests using Git. This is useful on more than one front because most devops will most likely be using git. Using a tool that will be useful in other areas, instead of a proprietary tool, is useful, even if readers end of using a Puppet Master instead of Git to distribute the manifests.

Don’t Take Shortcuts

This is a big one. Sometimes I’ll work on a project and near completion, only to rush it. It always winds up ruining the progress of the project, and the outcome is usually more work to get things right or a thing that doesn’t do what it was supposed to. Same goes for configuration management and managing everything as code. It’s important not to cheat. Once you trigger that configuration management tool, you shouldn’t be doing anything by hand that isn’t in a git repository or stored elsewhere. Don’t install shiny new packages by hand, use the configuration management tool.

It can be annoying and time consuming to hunt down bugs caused because one package was installed by hand and deploying the system breaks everything. The couple of extra seconds it takes to do it in an uncomfortable tool is well worth it. Not only does this help keep the debugging time down later, the more you use a new tool the more adept you’ll become with using it.

Story Time: GIT

When I was starting to get into advanced git usage, I started using my GUI client as a crutch because I didn’t want to spend the time to look up the command to do what I wanted. I wasted time getting the repository onto my computer just to update it with Sourcetree. One day I decided enough was enough and actually figured out how to accomplish the task1. Now every time I need to do that task, I can type a few keystrokes and it’s done. No more clone, modify, push cycle necessary.

Ultimately, writing everything as code is something that you have to decide if you want to commit to. If you’re not going to commit 100% to it, just set the boundaries. Know which situations you’ll want to use everything as code in and which you don’t. There is nothing wrong with that. There is something wrong with doing ad hoc everything and having no plan.

1: The actual task was selecting certain lines to commit. Using git add -p doesn’t work because it will only break a chunk down so far.
