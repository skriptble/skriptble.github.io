Javascript Ninja Skills The Core Languages

November 16, 2013

I struck lucky this week and received another 20% off coupon from Barnes and Noble. I am at the point where focusing on my core is important, so I’m buying books that help me focus. There is a Go book I’ve been eyeing for a while, but I’m determined to learn Go and most of Erlang using the resources of the web, not books. Multiple avenues of learning is going to be on this journey. That left me with a problem: Should I pass up a chance to save 30% on a book?

Skill Level: Ninja

Of course I didn’t pass up saving 30%. I have a habit of identifying books and thinking about them ahead of time. Now that I get newsletters from multiple publishers, I have an idea of the books I want before I get to the store. This week’s choice was a Javascript book co-authored by the creator of jQuery, Secrets of the Javascript Ninja. Javascript is one of the languages in my core, so I’m not deviating or cheating like I did last week.

Admittedly, I did pick up a couple of books on regular expressions. After some evaluation I decided that it made more sense for me to sharpen my Javascript skills. As a bonus the book I bought has a chapter dedicated to regular expressions. I see regular expressions as one of those auxiliary technologies I need to learn. Knowing six programming languages isn’t much use without the addition of other technologies.

I’m excited to have this book though. While web applications powered entirely by the server are cool, having sharp Javascript skills definitely makes a difference. As with most things, it’s not one or the other. Building great web applications requires both server-side and client-side code. Understanding the lower level languages is important, too. Dart 1.0 was released a couple of days ago and I read up on some of the chatter about it’s release. I tried it out about a year ago, but I didn’t end up getting involved in it. I wasn’t interested in front end development at the time.

Over time Dart might gain more traction over Javascript, but I still see value in learning vanilla Javascript. I’ve invested quite a bit of time in learning Javascript already. There’s also a chance it will help me learn Dart as well. Starting now it might make more sense to learn Dart, especially if you want to get to building things right away. From what I can tell, it’s there to make it easier to build web applications and it outperforms Javascript.

Coffeescript

While we’re on the topic of technologies that compile to Javascript, it makes sense to talk about Coffeescript. This language makes sense, because it adds a bunch of syntactic sugar to Javascript and helps make up for the places where Javascript comes up short. My core in learning the core isn’t to take the easy way and avoid the tough problems. Many say that Javascript is just a badly designed language, similar things are said about PHP. Everything begins the look like a bad design when we discover new designs in the world, unfortunately it isn’t as easy to simply adapt newer designs to older systems. Something I’ve learned is that every system is a legacy system. After that first line of code is written, everything going forward is a maintenance task.1 This means we can’t simply replace the badly designed thing with a better designed one. We should instead focus on improving the thing itself.

Coffeescript allows from Javascript to be written inside of it, and for the short term it doesn’t solve many problems. Similar to Dart, starting a brand new project with Coffeescript makes sense. Using Dart or Coffeescript also makes sense if a project is being rewritten from the ground up. That isn’t every project though. Just like C-extensions are written for PHP, sometimes the higher level language can’t accomplish the task at hand, and that doesn’t always pertain to performance, although sometimes it does.

jQuery

Another area that I am puzzled by at times is the use of jQuery. The web development community has a nasty case of framework-itis and library-itis. Everytime we wish to accomplish a task we throw in a new library or adapt a new framework. This creates lock in and sometimes kills performance. Libraries and frameworks have their place, but as happens with many tools (divitis anyone?) sometimes we over use the tool.

I want to explore the options of using vanilla Javascript to handle some tasks. It feels more targeted, and once again I find my Javascript skills lacking because I’ve been leaning on jQuery as a crutch. Understanding the underlying technology of a frame, library, or tool is more important than knowing how to use it. Uninformed developers may use a library like jQuery even though a native Javascript function will perform better and require less documentation.

I’m excited to be diving into Javascript at a deeper level. The skills I will gain from this will help me in other areas of my core. Between this book and Single Page Web Applications, I think I’ll be able to build some powerful front end applications.

1: A little something I picked up from The Passionate Programmer. Check it out, it’s a great book!
