We Built It Wrong

January 18, 2014

I’ve been reading quite a bit about hypermedia APIs, the support for them, and the arguments against them. I find the arguments against them perplexing actually, seeing as the thing they argue against is the same system that allowed their readers to actually find their argument. I’m not surprised though, we’ve had this issue for a while in the technology industry. Every time we come up with a new technology, we define it by the paradigms of what it is supposedly replacing. Instead of seeing architecture of web APIs as an extension of the web, we see them as extension of the internal systems programming we’ve been doing for quite a while. We are used to directly calling out a specific resource, and bundling resources together into packages. This isn’t how the web works, so trying to build systems that don’t utilize the web’s strength aren’t going to get the best results.

Fighting Is Difficult

Change does not come easy, especially when there is pride involved and people decidedly do things because that’s the only way they’ve known. I’ve experienced these types of things and have talked about it through various blog posts, but the worst side effect I’ve noticed is that the more “standard” the thing you’re trying to change, the more ridiculous backlash you’ll get. Hypermedia being used to power an application isn’t some ridiculous statement. The arguments against using hypermedia for an API don’t hold water. Mainly because the web itself is an API.

Disagree? Some might say “Well, it’s an API that’s only interfaced by humans”. This isn’t true at all. If that was true, the web would be much more difficult to use. When you look at it, Google is the API client of the World Wide Web. Obviously, Bing, Yahoo, and the other search engines are as well. Google isn’t written for a specific website, there aren’t different versions of Google for different websites. Google doesn’t need a custom client to pull data from Facebook or Twitter. The API client you’re using to read this blog post doesn’t either. Navigating to this website isn’t a singular interaction between you and the server. The browser client received the information from the server and did several more requests to load all the resources and display the final page to you. Facebook’s API, Twitter’s API, Google’s API, every API I’ve ever seen doesn’t follow this. I need a special client, written by that specific company, that specifically talks to their API.

We’ve built it wrong. Google did it right at first, because that’s the only way they could do it. Now things are wrong. They build their API’s so their own search engine API couldn’t crawl it1. The reason we did this is obvious: We didn’t want to change the way we built applications, nor did we have the server performance necessary for this to be accomplished. We have the possibility of server performance that can handle the amount of traffic necessary to make hypermedia APIs a reality. Now we just have to start building our applications in the same way we built the web.

On the web, things are much more flexible, redundant, and fault tolerant. Take DNS for example. If the main DNS servers that hold the master records broke, the entire internet doesn’t go down. That’s because DNS was built so that in order to be completely down, every DNS server in existence would have to go down. That isn’t going to happen2.

1: Okay so this might not actually be true, but you get the points.

2: Also not entirely true, large chunks of the internet could disappear if this happened.
